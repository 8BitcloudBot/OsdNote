# 路由在 RAG 与 Agent 系统中的作用

> 路由并非锦上添花，而是复杂系统的必然产物

---

## 一、为什么路由无处不在

任何复杂系统都绕不开一个基本矛盾：**单一能力单元覆盖不了所有输入**。

| 系统 | 没有路由的情况 | 路由解决了什么 |
|------|--------------|--------------|
| RAG | 所有问题都用向量库搜 | 事实性问向量库、计算性问 SQL、时效性搜网页 |
| Agent | 所有请求走一个通用 prompt | 写代码走编码 Agent、查天气调 API、搜索走浏览器 |
| 微服务 | 所有请求打到同一个服务 | 不同领域拆不同服务，API 网关做路由 |
| 数据库 | 单表存所有数据 | 分库分表 + 查询路由 |

路由本质上是一个**分解与分发决策**：识别当前输入属于哪个子空间，交给对应的子处理器。没有路由，所有逻辑就要塞进一个单体处理器里，引发复杂度爆炸和低内聚。

### RAG 中的典型路由

```
"最新的 iPhone 有什么功能？"         → 产品文档向量数据库
"我上次订购了什么？"                 → 用户历史 SQL 数据库（Text-to-SQL）
"A公司和B公司的投资关系？"           → 企业知识图谱（图查询）
```

### Agent 中的典型路由

```
"帮我写一个 Python 脚本"    → Code Agent（携带编码 tool 子集）
"今天杭州天气怎么样？"       → 工具 Agent（携带 API 调用 tool 子集）
"搜索量子计算最近进展"       → 搜索 Agent（携带浏览器 tool 子集）
```

## 二、路由准确性的三层保障

路由如果出错，后续所有处理都是无用功。

### 第一层：嵌入相似度路由（最稳）

分类数少（<20）、类别边界清晰时首选。

做法：
1. 每类预写描述文本 → 向量化并存储
2. 用户查询 → 向量化 → 算余弦相似度 → 取最近邻

优点：毫秒级延迟、结果确定性强、可设阈值兜底
缺点：类别边界重叠时混淆、描述质量直接影响效果

### 第二层：基于 LLM 的分类路由（最灵活）

分类数多、类别需要语义理解时使用。

做法：
1. 为每个选项配清晰互斥的描述 + few-shot 示例
2. LLM 分析查询并输出分类标签
3. 代码根据标签路由到对应处理链

最佳实践：
- 类别描述必须互斥
- 强制输出格式（"只能输出 A/B/C 之一"）
- 给 2-3 组 few-shot 示例
- 设计 "ambiguous" 类别作为兜底

弱点：长尾类别易遗忘、超过 10 类显著下降、延迟较高

### 第三层：确定性兜底（必须加）

无论上层用嵌入还是 LLM，兜底机制必不可少：
1. **置信度阈值** — 低于阈值走通用兜底
2. **ambiguous 分类** — 主动设计"不确定"路径
3. **路由结果二次校验** — 快速验证下游能否处理
4. **人工确认环** — 高价值场景："您是想查询产品信息还是订单状态？"

---

## 三、最佳实践

混合路由是最优解：嵌入路由做初筛（毫秒级）→ 不确定的走 LLM 分类 → 仍不确定的走兜底链路。

**核心原则**：与其让路由出错了硬着头皮往下传，不如在路由层就承认不确定。一个清晰的"我不知道该路由到哪"比一个错误的路由结果安全得多。
