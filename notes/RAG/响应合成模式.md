# 响应合成模式

> LlamaIndex 中 Response Synthesizer 的几种工作模式详解

---

## 一、背景

RAG 流程中，检索器召回了相关的文本块（Nodes），但不是简单拼接后丢给 LLM。响应合成器（Response Synthesizer）负责以不同的策略组织这些文本块和原始查询，再交给 LLM 生成最终回答。

不同模式本质是在回答一个问题：**多个文本块的信息如何整合进一次或多次 LLM 调用？**

---

## 二、核心模式

### 2.1 refne — 迭代优化

```
Node 1 + Query → LLM 生成"初始答案"
Node 2 + "初始答案" + Query → LLM 修正/补充 → "更新后的答案"
Node 3 + "更新后的答案" + Query → LLM 再优化 → "最终答案"
```

refne 中的"答案"指的是**逐步累加的中间回答**，不是最终答案。每处理一个 Node，"答案"就被 refne 一次，融合了之前所有 Node 的信息。

特点：
- 每个 Node 的信息都不会丢失（答案在累加）
- 调用次数 = Node 数量（成本高）
- LLM 有机会在后续轮次纠正早期错误
- 适合需要深度推理的场景

### 2.2 compact — 压缩调用

将所有 Node 的文本拼接，尽量塞进单次 LLM 调用的上下文窗口，超长时自动退化为 refne 模式分批处理。

特点：
- 理想情况一次调用完成，成本最低
- 依赖模型的上下文窗口大小
- 超长时退化到 refne，保证不会静默截断
- 适合大多数常规问答场景，是默认推荐模式

### 2.3 tree_summarize — 树状摘要

```
Node 1 + Node 2 → LLM → 摘要 A
Node 3 + Node 4 → LLM → 摘要 B
                  ↓
      摘要 A + 摘要 B → LLM → 最终答案
```

特点：
- 并行度最高（同层摘要可并发调用）
- 适合信息量大且需要分层归纳的场景
- Bottom-up 过程中可能丢失细节

### 2.4 simple — 简单拼接

直接将所有 Node 文本拼接后一次调用 LLM，无特殊策略。

---

## 三、模式对比

| 模式 | 调用次数 | 信息保留 | 延迟 | 适用场景 |
|------|---------|---------|------|---------|
| simple | 1 | 取决于上下文窗口 | 低 | 简单事实性问题 |
| compact | 1(理想) | 完整(不超长时) | 低 | 常规问答(默认推荐) |
| refne | = Node 数 | 完整且可修正 | 高 | 深度推理、精度优先 |
| tree_summarize | logN 层 | 逐层压缩可能丢失 | 中 | 大量信息需分层归纳 |

---

## 四、选择建议

- 不确定选哪个？用 `compact`。大多数情况下表现良好，超长时自动退化到 refne。
- 追求质量、不关心成本？用 `refne`。每个信息块被充分处理，且有机会修正。
- 检索结果非常多（几十个 Node）？用 `tree_summarize`，避免 refne 的线性成本爆炸。
